Email Account

Note: Our team relies heavily on Microsoft Outlook's Calendaring features in conjunction with Cisco's own Webex] application. To properly enable these feature, it is important that your email account is set up on the actual Exchange server (as opposed to a Scalix account). Contractors may need to specifically request an Exchange account.

  • About: Getting email.
  • Prerequisite: NT account.
  • Access: see instructions for access to different systems involved in this process
  • Instructions:
    1. Your manager should have also requested an email account for you. To check if you have an email account, you can go to Exchange Migration Status. If your User Dashboard does not appear, choose User > Search and enter your username. If this reports that you don't have an email account, contact your manager for assistance. If it reports that you have an email account but no Exchange account, you will have to continue and complete step 4. If you already have an exchange server set up, skip step 4 below.
    2. If you do have an account, go to http://pwreset.cisco.com and enter your NT username and password. Choose a new email password. Follow the password guidelines (Passwords must: be 8 characters, contain at least three of the following four types of characters: upper case characters (e.g. A-Z), lower case characters (e.g. a-z), digits (e.g. 0-9), "special" characters (e.g. !$%^&*()_+|~-=\`{}[]:";'<>?,./), must not be a word, and must not be based on personal info.)
    3. After successfully resetting your email password, you will receive a message saying "Your password has successfully been reset on _machine_". "_Machine_" is the name of your mail server.
    4. ---- possibly outdated information. please contact your manager if you do not have an Exchange account ---- Go to Exchange Messaging Availability where you can read about Exchange Account Provisioning, and follow a link to OnRamp if applicable, where you can make a request for an Exchange account (choose "Request/View Accounts"). This appears to apply only to contractors. Regular employees should be set up with an Exchange account by default. If you do need to request an Exchange account, you'll need to both request the account and verify your access to the mail server using an oddly formatted URL at the bottom of the page (AD Test Page). When clicking on the link titled "AD Test Page", be sure to enter your domain along with your user ID, for example: CISCO\yourid. (You can also use your email address "user@cisco.com", rather than the domain prefix/username combination). You need to both make the request, AND test the connectivity using the AD Test Page for the request to go through, but you can do this in any order you wish. After both steps have been completed, it may take up to 12 hours before your Exchange account is created. When the Exchange mailbox becomes active, mail sent by users with Cisco Exchange-based mail will INSTANTLY stop routing to your previous webmail box, so be aware of this. In order to see the mail from these users, use Exchange-based webmail: https://email.cisco.com/exchange/ . Mail from other users, such as external accounts will continue to come to your original webmail address, so you'll also need to check that as well (see below for configuring web-only webmail). In addition, you'll need to keep checking the Exchange-based Outlook account (set up automatically on a default-imaged machine) to see if it is active. Use the check name button in the account properties of your Exchange Outlook profile to verify the account. When you no longer get an error after checking your name, the account is ready to be accessed using Outlook. A mail will be sent from support when the Exchange mailbox is activated, but according to the support person I talked to, you may not see this confirmation mail in your webmail box...it might be routed to the new Exchange mailbox. (I saw mine...it came to the original webmail box about 2 minutes before the Exchange account became active.) Fun? YES! Confusing? YES! As a final indignity, all of your downloaded mail in your original webmail account will not be available in the new Exchange account. If you have set up an Outlook profile to check your original webmail, it will be contained in a .pst file. Import this .pst file into your new Exchange Outlook profile. "File>Import File>.pst File" your old mail file should be at: "C:\Documents and Settings\user\Local Settings\Application Data\Microsoft\Outlook\Outlook.pst"
    5. While you are waiting for your Exchange account to become active, you can setup your webmail client. Just be aware that this is a temporary endeavor. In Outlook, Eudora, or your preferred mail client, enter the following information:
      1. Your Name (self explanatory)
      2. Email address: username@cisco.com or firstname.lastname@cisco.com
      3. Username: your NT/email username
      4. Password (you just set this)
      5. Incoming mail server (probably a POP server): machine.cisco.com, where machine is the name from the password reset success message.
      6. Outgoing mail server (SMTP): mailman.cisco.com.
      7. When your Exchange account is ready, you might need to add the Exchange account in Outlook on your laptop. For newly imaged machines (as of 11/09), it appears that an Exchange account is setup correctly with your username by default.

Environment Testing

  • About: You now have Git installed, a clone of the Git repository, your CQ instance, Aptana/Eclipse set up, and the ability for Aptana to talk to CQ and vice versa using Vault, as well as the ability to upload your changes to the Git repository.
  • Prerequisite Git, repo checked out, CQ, Aptana, Vault
  • Access: Refer to other steps above
  • Instructions:
    1. In Aptana, open any .jsp file in your wem project (be sure you can locate the same file in CQ).
    2. Edit it by adding some text somewhere on the page. Save it.
    3. You will notice that it now has a star/arrow next to the file in the navigator. This tells you that this file is different than what is on the Git repository.
    4. Highlight the folder that this file belongs to, right click on it, and select VaultClipse? > Import (in Eclipse, you select Team > VaultClipse? > import).
    5. Vault commands only work on folders, not on individual files, so you do need to call the command from the closest folder that the file belongs to.
    6. The 'Import' command means that you are sending from Aptana to your CQ instance.
    7. Open your CQ/CRXDELite.
    8. Navigate to the same file, open it and see if the changes that you made are now appearing on the file in CQ too. If you see the changes, then the Import command worked as intended.
    9. In the open CQ file, make a change to the new text and save it.
    10. Go back to Aptana, and right click on the folder of that file again, and select VaultClipse? > Export.
    11. You should now see the change from the CQ file showing up in the same file in Aptana. This means that Vault is working correctly.

Sass

  • About: Sass is a CSS pre-processor. The wem project uses Sass to create conditional styling in Sass format and Sass converts it into standard CSS.
  • Prerequisite Ruby
  • Access: http://sass-lang.com/
  • Instructions:
